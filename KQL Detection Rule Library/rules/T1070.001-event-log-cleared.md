# T1070.001 — Indicator Removal: Clear Windows Event Logs

**Tactic:** Defense Evasion  
**Severity:** High  
**Data Source:** `SecurityEvent` (Windows Security Events connector)  
**Query Frequency:** Every 1 hour | **Lookback:** 24 hours

---

## Detection Rationale

Clearing Windows event logs is a classic anti-forensics technique used to destroy evidence of compromise. Legitimate log clearing is extremely rare and should always be change-controlled. This rule has very low baseline noise and is high-fidelity with minimal tuning.

---

## KQL Query

```kql
SecurityEvent
| where TimeGenerated > ago(1d)
| where EventID in (1102, 104)
| project TimeGenerated, Computer, EventID,
    SubjectUserName, Activity
| sort by TimeGenerated desc
```

**Event ID Reference:**
- `1102` — Security audit log was cleared
- `104` — System log was cleared

---

## Tuning Log

| Field | Detail |
|-------|--------|
| Baseline Noise | ~3 alerts/week |
| Tuning Period | 5 business days |
| Final FP Rate | ~0–1 alerts/month (outside change windows) |

**Exclusions Applied:**
- Excluded timestamps during documented server decommission change windows
- Excluded known server retirement admin accounts

**Analyst Notes:** Correlate with other defense evasion indicators in the same timeframe. Log clearing following suspicious activity (failed logins, process execution) significantly elevates severity.

---

## Response Actions

1. Identify who cleared the log and from which system
2. Check for an approved change ticket — if none exists, treat as incident
3. Review other log sources for activity in the window before the clear (Azure logs, firewall, DNS)
4. Determine what activity may have been concealed
5. If no legitimate explanation — initiate incident response
