# T1003.001 — OS Credential Dumping: LSASS Memory Access

**Tactic:** Credential Access  
**Severity:** Critical  
**Data Source:** `DeviceEvents` (Defender for Endpoint)  
**Query Frequency:** Every 1 hour | **Lookback:** 24 hours

---

## Detection Rationale

LSASS (Local Security Authority Subsystem Service) holds credential material in memory. Tools like Mimikatz, Procdump, and Task Manager can access this memory to extract NTLM hashes and Kerberos tickets. Any unexpected process opening LSASS is a critical signal requiring immediate investigation.

---

## KQL Query

```kql
DeviceEvents
| where TimeGenerated > ago(1d)
| where ActionType == "OpenProcessApiCall"
| where FileName =~ "lsass.exe"
| where InitiatingProcessFileName !in~ (
    "MsMpEng.exe", "csrss.exe", "werfault.exe",
    "taskmgr.exe", "svchost.exe", "lsass.exe")
| project TimeGenerated, DeviceName, InitiatingProcessFileName,
    InitiatingProcessCommandLine, AccountName
```

- Built allowlist from 2-week baseline observation

**Analyst Notes:** Any process not on the allowlist accessing LSASS is critical. Isolate the endpoint while investigating — do not wait for confirmation.

---

## Response Actions

1. Immediately isolate the endpoint via Defender for Endpoint
2. Identify the initiating process — is it signed? Known? Expected on this system?
3. Assume credential compromise — initiate password reset for all accounts that logged into the system
4. Run full forensic collection before remediation
5. Hunt for lateral movement using the potentially dumped credentials
