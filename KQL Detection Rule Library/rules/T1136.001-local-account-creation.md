# T1136.001 — Create Account: Local Account (After Hours)

**Tactic:** Persistence  
**Severity:** Medium  
**Data Source:** `SecurityEvent` (Windows Security Events connector)  
**Query Frequency:** Every 1 hour | **Lookback:** 24 hours

---

## Detection Rationale

Creation of local user accounts outside standard business hours may indicate attacker-established backdoor accounts or insider threat activity. Low baseline volume makes this rule inherently high-fidelity after minimal tuning.

---

## KQL Query

```kql
SecurityEvent
| where TimeGenerated > ago(1d)
| where EventID == 4720
| extend HourOfDay = datetime_part("Hour", TimeGenerated)
| where HourOfDay !between (8 .. 18)
| where dayofweek(TimeGenerated) !in (0, 6)
| project TimeGenerated, Computer, SubjectUserName, TargetUserName, HourOfDay
```

---

## Tuning Log

| Field | Detail |
|-------|--------|
| Baseline Noise | ~5 alerts/day |
| Tuning Period | 5 business days |
| Final FP Rate | ~0–1 alerts/week |

**Exclusions Applied:**
- Excluded documented admin accounts performing after-hours work (change ticket correlation)
- Excluded automated provisioning service account

**Analyst Notes:** Every alert should be investigated. Correlate with Event ID 4722 (account enabled) and 4732 (added to Administrators group).

---

## Response Actions

1. Identify who created the account (`SubjectUserName`) and on which system
2. Check if a change ticket exists for the activity
3. Review account privileges — was it added to local Administrators?
4. Disable and investigate if creation is unexplained
5. Hunt for the account on other systems in the environment
